## What was built

A browser 3D physics playground in `source/src/main.ts` — 50 procedurally-generated cubes cascade onto a tilted ramp, collide, and pile up. Verified working via headless Chrome (screenshots above): a settled state showing the pile with **37 live contacts**, and a mid-fall state after pressing **Reset** showing cubes dropping with **0 contacts**.

**Architecture** — built on Aura3D's own subsystems:
- **Simulation:** `@aura3d/engine/physics` — the library's real rigid-body engine (`PhysicsWorld`, `Shape`, `RigidBody`): gravity, broadphase, contact detection, collision response, friction, and sleeping. The ground and tilted ramp are static infinite planes; the 50 cubes are dynamic box bodies.
- **Rendering:** `three` (the renderer the Aura3D engine itself ships and uses) + `OrbitControls` for camera orbit. Meshes are synced each frame from physics body transforms, with shadows.
- **Overlay:** HTML panel with title, **live contact count** (read from `world.snapshot().stats.contacts`), cube count, settled count, FPS, and a **Reset Simulation** button.

**Required visual evidence — all satisfied:**
- Many cubes (50, colored) ✓
- Tilted ramp with side rails ✓
- Collision/contact behavior — visible pile + live contact count (peaks at 49, settles to ~30) ✓
- Reset control + live contact overlay ✓
- Orbiting camera framed so the setup reads at a glance ✓

## Assumptions

- **Prompt feasibility vs. the declarative API.** Aura3D's headline declarative API (`createAuraApp`, `scene()`, etc.) renders a *static* snapshot with no per-frame mutation hook, so it cannot express a live physics simulation, reset, or changing contact count on its own. I therefore built on the engine's deeper, exported subpaths: `@aura3d/engine/physics` for the actual simulation and `three` (the engine's own rendering dependency) for drawing. These are part of the `@aura3d/engine` package's `exports` map, not third-party additions — `three` arrives transitively via the engine, no new direct dependency was added.
- **Ramp modeled as a tilted infinite plane** (with a horizontal floor plane) rather than a finite slab, because the engine's box collider uses an axis-aligned bound that ignores rotation — a tilted *plane* gives correct on-slope resting/sliding. A matching finite slab mesh is drawn for the visual ramp.
- Friction (~0.5) slightly exceeds tan(22°), so cubes mostly settle into a pile on the ramp rather than sliding entirely to the base — this keeps a dense, clearly-visible contact cluster.
- `package-lock.json` was generated by `npm install` (expected). `devDependencies` remain the original `typescript` + `vite` only.
