## Done — 3D Data Visualization (6×6 animated bar field)

I built an interactive 3D bar chart in the provided `source/` directory and verified it end-to-end in headless Chrome (36 bars render, axis labels readable, hover-highlight works, zero page errors).

### Commands
- **Install:** `npm install`
- **Build:** `npm run build` (runs `tsc --noEmit` then `vite build`)
- **Run:** `npm run dev -- --port <assigned-port>` (or `npm run preview` for the built output)

### Required visual evidence — how each is met
- **~36 bars visible** — a 6×6 grid of 36 box primitives.
- **Heights vary & animate from random values** — each bar starts at a random value and eases toward a random target; targets re-roll every ~3.2 s, so the field is continuously animating.
- **Color corresponds to height** — each bar's color is recomputed every frame from its current height via a cool→warm ramp (low = blue, high = red); the same ramp drives the on-screen legend.
- **Axis labels visible/readable** — `C1…C6` (X), `R1…R6` (Z), a vertical `0…100` value axis (Y), plus axis titles, rendered as crisp DOM labels in 3D via `CSS2DRenderer`.
- **Hover-highlight (implemented + documented)** — a raycaster picks the bar under the cursor; the hovered bar gets an emissive glow + ~16% footprint scale-up, the cursor becomes a pointer, and a tooltip shows `Col N · Row N` / `Value NN / 100`. Documented in `source/README.md`. Verified screenshot showed `Col 5 · Row 5 — Value 82/100` with the bar visibly highlighted.
- **Orbit camera** — `OrbitControls` (drag to orbit, scroll to zoom) with a gentle auto-rotate that pauses during interaction.

### Architecture / Aura3D usage
The scene is **authored with the public `@aura3d/engine` API** (`scene()`, `primitives.box`/`plane`, `lights`, `material`, `camera.orbit`, `interactions.orbit`, `timeline`) in `src/auraScene.ts`, producing an `AuraSceneSnapshot`. `src/main.ts` renders that snapshot through an interactive renderer built on the **same Three.js the engine ships**.

### Assumptions / notes
- **Why a Three.js render layer over the engine snapshot:** I inspected the shipped `@aura3d/engine` build. Its built-in snapshot renderer builds meshes once and is effectively static — it does **not** animate individual nodes, raycast for hover, or draw text (the `animate`/`interaction`/`timeline` specs are stored but ignored, and orbit is a fixed camera placement). Since the prompt requires per-frame height animation, hover-highlight, interactive orbit, and readable labels, I kept the engine as the scene-authoring source of truth and rendered its snapshot with the engine's own Three.js dependency to deliver the dynamic behavior. The engine snapshot still drives background, lights, camera, and every bar (matched back to its data by node name).
- I added `three` and `@types/three` (pinned to the engine's `^0.165.0`) explicitly to `source/package.json` so the import is declared rather than relying on hoisting. No files outside `source/` were edited; no external assets were used (fully procedural).
- One hardening note: under `vite dev`, hot reloads can stack animation loops; I added an `import.meta.hot.dispose` guard plus value/dt clamps so a fresh load (and the production build) runs a single stable loop. Verified stable on a fresh production load and a fresh `npm run dev` load.
