## What was built

A studio **product viewer** for the provided `sneaker.glb`, written entirely against the public `@aura3d/engine` API.

**Files (all inside `./source`):**
- `src/aura-assets.ts` — typed asset module via the public `defineAuraAssets`, pointing at the only allowed asset, served from `public/benchmark/assets/sneaker.glb` at URL `/benchmark/assets/sneaker.glb`. No invented paths.
- `src/main.ts` — composes the scene: `prefabs.productStage()` (plinth + backdrop + softbox cards + contact shadow), the sneaker `model(assets.sneaker)` with a continuous turntable spin (`.animate({ clip: "turntable" })`), a multi-light studio rig (ambient + studio key + cool key / front fill / warm rim points), subtle bloom, `interactions.orbit()`, and an orbit camera framed on the product.
- `index.html` — full-viewport canvas styling.

The engine auto-centers and fit-to-bounds auto-scales the GLB, so the sneaker always sits centered on the plinth.

## Assumptions
- The provided `sneaker.glb` lives at `source/public/benchmark/assets/sneaker.glb`, so Vite serves it at `/benchmark/assets/sneaker.glb` — the URL the typed asset references. This is the provided path, not an invented one; I did not run the network-dependent `@aura3d/cli assets add`, hand-authoring the same typed-asset module instead so the asset stays at its provided location.
- Aura3D's `interactions.orbit()` + orbit camera express orbit-controls declaratively, and `.animate` drives the turntable in the engine's render loop; I kept everything on the public engine API rather than dropping to raw Three.js, per the project instructions.
- Diagnostics overlay is left on (corner HUD) as evidence the GLB loaded; it does not obscure the product.
