## Summary

Built a procedural **Physics Playground** in raw Three.js 0.165.0 (Vite + TS strict) at `source/src/main.ts`. No external physics engine — a compact self-contained rigid-body sim handles it:

- **50 falling cubes** (colored `MeshStandardMaterial` boxes) cascading from a column above the ramp.
- **Tilted ramp** — a 24° slab rendered as a rotated box with a matching oriented-bounding-box collider.
- **Collision response** — gravity + substepped integration; sphere-vs-OBB collisions (closest-point in box-local space) against ramp, ground and translucent retaining walls, plus O(n²) sphere-vs-sphere between cubes, all with restitution + friction and contact-driven angular damping.
- **Camera orbit** — `OrbitControls` (damped) framed on the setup; drag to orbit, scroll to zoom.
- **DOM overlay** — title, cube count, **live contact count** (recomputed each frame), FPS, and a **Reset** button that re-scatters the cubes.

Added `@types/three@0.165.0` to the scaffold's devDependencies so strict `tsc --noEmit` passes.

### Commands
- **Install:** `npm install`
- **Build:** `npm run build`  *(runs `tsc --noEmit` then `vite build` — passed)*
- **Run (for the runner):** `npm run dev -- --port <assigned-port>`

### Assumptions
- This is the raw Three.js benchmark track (per `context/llms.txt`), so I used `three` + bundled `three/addons` only and implemented the physics myself — the "don't build a custom physics engine" rule applies to the Aura3D track, which exposes no physics API here.
- Cubes are approximated as inscribed collision spheres (radius = half edge) for tractable, stable contacts; render geometry is full boxes that visibly rest, slide, and pile on the ramp.
- All work is confined to the `source/` directory.
