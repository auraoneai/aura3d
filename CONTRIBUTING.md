# Contributing Guide

Version: 1.3.2

## Scope

Contributions for version `1.0.5` must preserve the current governance rules:

- keep implementation edits scoped to the owning package or tool;
- do not revert unrelated work from other agents or contributors;
- add focused tests for behavior changes;
- update source checkboxes only after the relevant command passes;
- keep public claims aligned with `docs/project/product-studio-claim-registry.md`.

## Local Verification

Run the smallest relevant command first, then broader gates when appropriate:

```sh
pnpm typecheck
pnpm test:unit
pnpm test:integration
pnpm verify:claims
pnpm verify:release
```

For governance-only changes, run the governance unit tests and claim scan:

```sh
pnpm exec vitest run tests/unit/tools/governance-docs.test.ts
pnpm verify:claims
```

## Pull Request Checklist

- Describe the changed paths and owning workstream.
- List commands run and whether each passed or failed.
- Link report JSON files generated by the change.
- Explain any remaining blockers.
- Do not use stronger public claim language than `docs/project/product-studio-claim-registry.md` permits.

## Issue Templates

Use `.github/ISSUE_TEMPLATE/bug_report.yml` for defects and `.github/ISSUE_TEMPLATE/feature_request.yml` for enhancements.


## Good First Issues

Good first contributions include:

- improving agent docs, prompt recipes, and typed asset examples;
- adding Vite template documentation for browser 3D apps;
- tightening comparison wording while preserving claim boundaries;
- adding screenshot or route-health evidence to examples;
- improving troubleshooting docs for blank canvas, missing GLB/glTF assets, Vite deploy paths, and screenshot failures.

## Docs And Example PRs

For docs or example pull requests, include:

- the source prompt or use case;
- asset source and license when a GLB/glTF file is involved;
- commands run and whether each passed or failed;
- screenshot path or route-health evidence when visuals changed;
- any claim boundary that applies, especially for manual renderer code comparisons or benchmark wording.

## Issue Reports

When filing an issue, include package version, template or route, prompt, asset source, commands run, runtime error, route-health output, screenshot evidence, and deployment target when relevant.
