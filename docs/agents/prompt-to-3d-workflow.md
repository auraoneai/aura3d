# Prompt To 3D Workflow

Prompt-to-3D TypeScript workflow for AI coding agents. Use this for AI-generated browser 3D scenes, scene kits, typed GLB/glTF assets, diagnostics, screenshots, and deployment checks.

Version: 1.0.0

Aura3D is an agent-friendly browser 3D SDK. The expected output is normal
TypeScript or JavaScript that uses `@aura3d/engine`, typed assets generated by
the Aura3D CLI, and route-level evidence that a human team can inspect.

This document is the durable agent-facing version of the scoped product-context
arbitrary prompt will beat low-level renderer code.

It is also not a provider-runtime prompt-to-scene contract. In the active product, the AI coding agent turns prompt intent into editable TypeScript or JavaScript source code that calls public Aura3D APIs.

## Aura3D advantage

Aura3D provides:

- public scene, asset, camera, lighting, material, effect, interaction,
  physics, chart, city, character, product, UI, and scene-kit helpers;
- typed GLB/glTF asset references generated by `@aura3d/cli`;
- prompt-plan helpers for turning scene intent into maintainable source code;
- scene kits for common benchmark and product prompt families;
- diagnostics, screenshot evidence, route-health checks, and deployment checks;
- package and documentation artifacts that can be reviewed before release.

Aura3D does not provide:

- a hidden asset store;
- an LLM-generated binary scene format;
- a provider-backed prompt-to-IR runtime as the current public product;
- permission to invent asset URLs or string ids;

## Default Workflow

1. Restate the requested 3D experience as scene intent.
2. If you do not already have the asset file, resolve it from the catalog: run `npx @aura3d/cli@latest assets search "<intent>"`, pick an auto-pullable candidate (CC0 / CC-BY, verified, direct-download), and let the CLI pull and type it. Everything else is a deep-link you open and license yourself. Then use `model(assets.x)` with the generated key. Do not invent a URL or fall back to a primitive for a named real object.
3. Add a real asset you already have on disk with `npx @aura3d/cli@latest assets add ./assets/model.glb --name model`.
4. Import `assets` from `./src/aura-assets` or the generated relative path for the app.
5. Start from `definePromptPlan(...)` or the closest `sceneKits.<name>()` helper.
6. Add only prompt-required customization.
7. Enable diagnostics or collect scene evidence when the output will be reviewed.
8. Run finite build and check commands appropriate to the app.
9. Use screenshots, route health, typed asset provenance, and deployment checks as evidence.

## Typed Asset Rule

Do this:

```ts
import { createAuraApp, lights, model, scene } from "@aura3d/engine";
import { assets } from "./aura-assets";

createAuraApp("#app", {
  scene: scene().add(model(assets.model)).add(lights.studio())
});
```

Do not do this in safe API examples:

```ts
model("model");
model("/assets/model.glb");
model("https://example.com/model.glb");
```

Use `unsafeModelUrl(...)` only as an explicit temporary escape hatch outside
safe benchmark or release-facing examples.

## Prompt-Plan Pattern

Use a prompt plan when the user gives product intent rather than explicit scene
code.

```ts
import {
  compilePromptPlan,
  createAuraApp,
  definePromptPlan,
  promptPlanToScene
} from "@aura3d/engine";
import { assets } from "./aura-assets";

const plan = definePromptPlan({
  sceneType: "product-viewer",
  subject: { asset: assets.product },
  camera: { preset: "product-orbit" },
  lighting: { preset: "studio-softbox" },
  effects: ["bloom"],
  interaction: "orbit",
  acceptanceCriteria: [
    "product is centered",
    "studio lighting and contact shadow are visible"
  ]
} as const);

const compiled = compilePromptPlan(plan);

createAuraApp("#app", {
  scene: promptPlanToScene(plan),
  diagnostics: { overlay: true }
});

console.log(compiled.report.visualSystems, compiled.report.repairHints);
```

A compiling app is not enough. The screenshot must visibly satisfy the prompt.
If the screenshot is weak, apply the repair hints before changing the review
label or public claim.

## Scene-Kit Selection

Use scene kits when the prompt maps to a maintained visual system.

| Prompt family | Preferred API | Evidence expected |
| --- | --- | --- |
| Physics playground | `sceneKits.physicsPlayground()` | falling cubes, ramp, pile, contact cues, reset affordance |
| Particle fountain | `particleFountain(...)` from `@aura3d/engine/scene-kits/particle-fountain` for the benchmark recipe | dense muted droplets, nozzle, splash context, real emission-rate control |
| Solar system | `sceneKits.solarSystem()` | sun glow, six labeled planets, orbit paths, stars, depth framing |
| Neon tunnel | `sceneKits.neonTunnel()` | inside-tunnel camera, receding rings, rails, reflection, fog, controlled bloom |
| Data visualization | `sceneKits.dataViz({ dataset })` | bars, axes, ticks, legend, title, selected value, no orphan labels |
| Mini-golf | `sceneKits.miniGolf()` | ball, cup, aim, power, score, obstacle, boundaries, follow-camera cue |
| Material lab | `sceneKits.materialLab()` | metal, glass, rubber, emissive, clearcoat, contact shadows |
| City block | `sceneKits.cityBlock({ timeOfDay })` | buildings, streets, crosswalks, props, traffic or street lights, day/night state |
| Humanoid walk | `sceneKits.humanoidWalk({ animationState: "benchmark-pose" })` | connected humanoid, planted feet, face cues, motion/path evidence |
| Product viewer | `sceneKits.productViewer(assets.product)` | typed model, plinth, contact shadow, studio softboxes, orbit/turntable cue |

before using this general workflow. The benchmark recipes include the current
lean Prompt 02 particle-fountain subpath and Prompt 09 humanoid-walk subpath.

## Evidence To Capture

Use these evidence surfaces for product work:

- `collectAuraSceneEvidence(scene)` for scene structure and feature metadata.
- `compilePromptPlan(plan).report` for visual systems, negative criteria, and repair hints.
- route-health reports for readiness, backend, draw calls, runtime errors, and diagnostics.
- screenshot artifacts for human visual inspection.
- typed asset manifests for provenance and anti-hallucination checks.
- `npx @aura3d/cli@latest check-deploy` for deployability checks.

Do not use these as final benchmark proof:

- nonblank screenshot checks;
- object counts;
- node names such as `bloom` or `contactShadow`;
- self-authored visual QA scores;
- diagnostics showing requested effects without accepted pixels;
- owner-scoped bypass notes for neutral review or external scoring.

## Product App Acceptance

A product app can be treated as scoped product-context evidence when:

- the app uses public Aura3D imports;
- assets are typed and generated by the CLI;
- build and route-health checks pass;
- screenshots show the requested scene rather than a symbolic sketch;
- diagnostics or evidence reports identify the scene systems used;
- public wording stays within the Aura3D SDK claim.

benchmark process in `docs/project/frozen-benchmark-release-gates.md` passes.
