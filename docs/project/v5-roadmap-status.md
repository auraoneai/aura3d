Historical status: superseded by V9.

V5 is retained as historical planning and evidence context. Its original completion rule required `pnpm three-compat:release` and the completion audit to prove every named requirement in `docs/project/three-compat-roadmap-visual-engine-plan.md`; current release/claim decisions now use the V9 docs.

V4 completed a bounded V4 SDK/product suite. That is the baseline, not the destination.

V5 must build a visually undeniable, developer-usable, broad Three.js replacement track for documented mainstream browser 3D use cases. It must not claim full Three.js API parity, Unity replacement, Unreal replacement, full WebGPU parity, broad performance superiority, or total ecosystem replacement unless future evidence explicitly supports those claims.

The first release-blocking work is:

- create the V5 truth/progress gates
- preserve deleted legacy/demo-file cleanup
- inventory the installed Three.js baseline
- build a compatibility matrix before building new screenshots
