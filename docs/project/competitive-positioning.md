# Competitive Positioning

Version: 1.0.0

A3D is a TypeScript-first browser 3D engine and workflow SDK. It competes on first-party browser 3D workflows, diagnostics, package boundaries, and route/report evidence.

## Positioning Statement

A3D gives web teams a first-party 3D stack with product-viewer, material-review, asset-inspection, animation, physics, WebGPU/WebGL2, WebXR, migration, and renderer-diagnostic workflows.

Current defensible wording:

> A3D has first-party browser 3D packages and current generated evidence for measured feature and visual parity slices against tracked Three.js workflows. Broader superiority claims require the full Three.js superiority gate to be regenerated and passing.

## Versus Three.js

Three.js remains the ecosystem benchmark. A3D differentiates through:

- workflow-level APIs for product, asset, material, scene, animation, physics, WebGPU/WebGL2, and migration use cases;
- route and report diagnostics for assets, renderer state, animation motion quality, resources, and visual review;
- first-party package boundaries across renderer, assets, animation, controls, materials, environments, input, physics, workflows, product-studio, debug, and Three.js compatibility helpers;
- current report generators that map feature coverage, visual quality, performance, animation, physics, memory, developer workflow, and claims to concrete artifacts.

## Versus Babylon.js

Babylon.js is a mature engine-style web 3D framework. A3D's wedge is workflow-first browser delivery with package-level diagnostics and reportable evidence. The current Babylon comparison is benchmark-slice evidence, not broad Babylon parity.

## Versus Unity And Unreal

Unity and Unreal target broader editor/native/game-engine markets. A3D should be positioned as browser-native workflow infrastructure, not as a Unity or Unreal replacement.

## Target Users

- Frontend and graphics teams building product viewers and configurators.
- Ecommerce, industrial design, digital twin, and internal-tool teams needing repeatable GLB/HDR/PBR previews.
- Web teams migrating selected Three.js workflows into package-owned APIs.
- Engineering teams that need browser screenshots, route health, asset diagnostics, and resource lifecycle proof.
