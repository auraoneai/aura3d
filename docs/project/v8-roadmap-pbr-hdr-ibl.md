# V8 PBR, HDR, And IBL

> Historical note: This V8 document is retained as project history after the V9 parity reset. Current planning, claim boundaries, and code-backed parity status live in `docs/project/v9-roadmap-status.md`, `docs/project/v9-roadmap-parity-matrix.md`, and `docs/project/v9-roadmap-three-js-parity-plan.md`. Treat unchecked tasks or old claims here as historical unless they are restated in the V9 docs.


V8 requires high-fidelity PBR/HDR/IBL output, not gray-box or Lambert-like proof frames.

## PBR Material Model

The renderer must provide physically based material behavior for real imported assets:

- base color
- metallic and roughness
- normal maps
- emissive color and strength where supported
- alpha modes
- clearcoat, sheen, specular, transmission, volume, anisotropy, and iridescence only when proven by route evidence

Unsupported material extensions must be reported in diagnostics rather than silently ignored as if parity exists.

## HDR Environment

HDR environment setup must support:

- loading real HDR assets
- applying them as visible lighting inputs
- exposing selected environment metadata
- reporting failures visibly
- switching environments without rebuilding proof-only app state

The environment should affect reflections and diffuse lighting, not merely appear as a background label.

## PMREM

PMREM or equivalent prefiltered environment lighting is required before claiming high-end PBR/IBL parity. The claim boundary is:

- Allowed: "A3D has route-level PMREM/IBL evidence" only where comparison reports prove it.
- Blocked: "A3D matches Three.js PMREM globally" until the same-scene comparison set covers the relevant materials and environments.

## Tone Mapping

Tone mapping must be explicit and configurable. The current target is ACES-like output with exposure controls and stable sRGB presentation.

Required diagnostics:

- tone mapping mode
- exposure
- output color space
- environment id
- backend

## Shadows And Postprocess

V8 visual proof should include shadows and postprocess only when they improve product output and are represented honestly in diagnostics.

Blocked claims until proven:

- full cascaded shadow quality parity
- full postprocess stack parity
- WebGPU feature parity with WebGL2
- physically exact offline-renderer parity

## Visual Acceptance

PBR/HDR/IBL screenshots must pass metric gates and human notes in `tests/reports/current-routes-visual-review.json`. A screenshot can fail even if it is nonblank, high resolution, and generated by the real renderer.
