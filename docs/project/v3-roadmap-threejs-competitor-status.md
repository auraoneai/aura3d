# Three.js Competitor Status

> Historical note: This V3 document is retained as project history after the V9 parity reset. Current planning, claim boundaries, and code-backed parity status live in `docs/project/v9-roadmap-status.md`, `docs/project/v9-roadmap-parity-matrix.md`, and `docs/project/v9-roadmap-three-js-parity-plan.md`. Treat unchecked tasks or old claims here as historical unless they are restated in the V9 docs.


V3 includes a local same-scene comparison against Three.js for four workflow scenes:

- Product scene.
- Material scene.
- glTF asset scene.
- Interactive scene.

The comparison evidence is generated by:

```sh
pnpm v3:compare-threejs
```

The report is written to:

```text
tests/reports/foundation-threejs-comparison.json
```

Screenshot evidence is written under:

```text
tests/reports/foundation-threejs-comparison/
```

Each scene has:

- G3D screenshot.
- Three.js screenshot.
- Diff screenshot.
- Setup-line count.
- Runtime draw-call data.
- Feature notes.
- Gap list.

## Current Result

The current evidence supports this claim:

> G3D is a credible Three.js competitor for the tested supported workflows because the workflow APIs produce comparable local screenshots with less setup code and built-in diagnostics.

## Claim Boundary

The evidence does not support these claims:

- Broad Three.js replacement is not proven.
- Broad Three.js superiority.
- Full Three.js ecosystem parity.
- Full glTF loader parity.
- Full WebGPU parity is not proven.
- Full game engine parity.

Three.js remains stronger in ecosystem size, plugin breadth, community examples, low-level scene graph flexibility, and production battle testing.
