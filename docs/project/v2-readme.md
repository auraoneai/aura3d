## Purpose

This v2 documentation set restates Aura3D's current feature evidence and the work required before external claims such as "better than Three.js" or "competitive with Unity/Unreal for the web" are credible.

The existing `docs/*.md` corpus is a rebuild specification, PRD set, trace system, and generated audit record. It is useful, but it mixes product ambition, internal gate status, generated trace evidence, stale gap language, and completion claims. The v2 docs are intentionally different:

- They separate current implementation evidence from external-market claims.
- They treat internal tests as necessary but not sufficient.
- They define what must be proven with real applications, real assets, real browsers, and comparative benchmarks.
- They avoid using generated trace rows as the sole proof of product readiness.

## Verification Status Policy

No prose snapshot in this directory is authoritative. Verification status must come from report files generated by the same release-run ID, from a clean checkout, with the commands and environment recorded in the report.

Required rules:

- Do not call a report "latest" unless its run ID matches the current release verification run.
- Do not close a gate from a stored green JSON report if any source, test, example, benchmark, or docs file changed afterward.
- Do not copy transient local failures into long-lived docs as permanent truth; link to report files and regenerate them.
- Do not make public claims from internal trace completion alone.

As of the current v2 evidence review, the project still lacks repeated clean-checkout release evidence, a broad Blender/exporter corpus, a broad better-than-Three.js claim, externally hosted demos, a versioned package release, and independent reproduction evidence. App-scale local demos, comparison reports, public docs, a bounded browser/hardware matrix, known-limits enforcement, bounded KTX2/Basis texture transcoding, three checked-in Blender-export fixture validations, and an exact checked-in scaffold bundle-size niche now exist, but they remain local or bounded evidence and do not enable broader external claims by themselves.

## How To Execute The v2 Plan

Use [Filename-Level Execution Checklist](./filename-level-execution-checklist.md) as the implementation control document. Every task should become an issue or PR with:

- the exact file paths it changes;
- the exact tests it adds or updates;
- the examples or apps it proves;
- the report JSON files it produces;
- the claim gate it advances.

Do not close an external claim from a prose update. Close it only from the checklist, tests, examples, reports, and decision gates.

## Status Language

Use these terms consistently:

| Status | Meaning |
|---|---|
| Internal slice exists | Code and tests cover a bounded implementation path. |
| Internally verified | The current repo gates prove the path under controlled conditions. |
| Externally credible | Real-world assets, browsers, devices, examples, docs, and comparative benchmarks support a public claim. |
| Production credible | External apps, regression history, compatibility matrix, documented limits, and support process exist. |

Aura3D currently has many internal slices and one exact checked-in scaffold bundle-size niche. It is not yet externally credible as broadly "better than Three.js" or "Unity/Unreal for the web."
