# Old Codebase Port Plan

> Historical note: This V4 document is retained as project history after the V9 parity reset. Current planning, claim boundaries, and code-backed parity status live in `docs/project/v9-roadmap-status.md`, `docs/project/v9-roadmap-parity-matrix.md`, and `docs/project/v9-roadmap-three-js-parity-plan.md`. Treat unchecked tasks or old claims here as historical unless they are restated in the V9 docs.


This plan defines how to mine the older `master` branch for useful visual and workflow features, adapt them into the current V3/V4 architecture, and verify that the result works in-browser before any claim is expanded. It is a planning artifact only; no feature is considered ported by this document.

Audited current branch: `preserve/a3d-v2-execution-state`.
Audited old branch: `master` in the same repository.
Primary rule: port behavior and evidence, not old API shape.

## Executive Summary

The old branch contains useful showcase material for procedural cars, racing tracks, procedural textures, starfields, architectural scene composition, postprocess controls, and shader math references. It also contains risky renderer snapshots (`Renderer.ts.bak9` through `.bak12` and `.corrupt`) and broad old-engine API surfaces that should not be copied directly.

The best path is an evidence-driven port:

1. Add missing geometry primitives in the current renderer (`cylinder`, `capsule`) with unit coverage.
2. Port deterministic procedural texture fixtures and use them as local, license-safe material assets.
3. Build a current-engine racing showcase from the old car, track, HUD, and race-loop concepts.
4. Add deterministic starfield and nebula backgrounds to game/postprocess surfaces.
5. Apply texture and environment upgrades to product, game, material showroom, and architecture scenes.
6. Rebuild architecture visual composition using the old room/furniture/material/lighting concepts.
7. Expand the postprocess setting model only where real pixel-tested effects exist.
8. Use old PBR/shadow chunks as references for current shader work, with browser pixel gates.

This is a route toward stronger Three.js/Babylon-style example quality. It does not, by itself, prove Unity or Unreal parity. Unity/Unreal claims remain blocked until the editor, import, authoring, runtime, lighting, asset workflow, deployment, and profiling gates all pass with comparable evidence.

## Claim Policy

The port must preserve the current V3/V4 claim discipline.

- Do not claim Three.js superiority unless a same-scene, same-feature, same-viewport comparison report proves the exact measured niche.
- Do not claim Babylon.js superiority unless the same comparison standard passes.
- Do not claim Unity replacement, Unreal replacement, or Unity/Unreal parity from example visuals alone.
- Do not claim production readiness from generated assets, procedural scenes, or isolated visual demos.
- Do not claim full PBR, production shadows, HDR, postprocess suite parity, or game-engine parity unless the relevant V4 decision gate permits that exact wording. Full glTF and full WebGPU wording must follow their current readiness reports.
- Every unsupported imported old feature must be visibly blocked in code state, UI/report evidence, docs, and claim-gate tooling.
- Broad claims remain blocked even if screenshots look better.

Allowed language after successful phases should stay narrow:

- "Aura3D includes a bounded current-engine racing showcase with procedural local assets and browser screenshot evidence."
- "Aura3D uses deterministic local procedural texture fixtures in V4 examples."
- "The V4 architecture viewer includes richer generated composition and verified material/light preset state."
- "Specific postprocess operators are pixel-tested on real WebGL2 scene readback."

## Visual Quality Gate Requirements

Before any ported feature is marked complete, the gates must fail the following cases:

- Raw JSON/debug UI dominating the screenshot.
- Primary asset too small, off-screen, or clipped.
- Dark/blank/corrupt screenshot.
- Untextured primitives claiming showcase quality.
- UI text overlaps or does not fit on desktop/mobile.
- Stale screenshot path or report from a previous commit.
- Metadata-only evidence without browser pixels.
- Feature evidence that stays true after the visual feature is disabled.
- Same-scene comparison with different camera, viewport, asset class, feature state, or unsupported-feature handling.

Recommended gate extensions:

- Extend `tests/browser/example-screenshot-audit-external-parity.spec.ts` for each new example and each new required evidence key.
- Extend `tools/external-parity-visual-quality/index.ts` with per-example minimum nonblank pixels, color buckets, primary asset area, and not-debug-dominated checks.
- Add a contact sheet artifact under `tests/reports/v4-example-screenshots/` for manual review, but do not let manual notes override failed automated gates.
- Add negative fixture screenshots or synthetic manifest entries so `pnpm verify:external-parity-visual-quality` proves known-bad cases fail.
