# A3D V6: Production Renderer, Real Assets, True PBR/HDR Parity

> Historical note: This V6 document is retained as project history after the V9 parity reset. Current planning, claim boundaries, and code-backed parity status live in `docs/project/v9-roadmap-status.md`, `docs/project/v9-roadmap-parity-matrix.md`, and `docs/project/v9-roadmap-three-js-parity-plan.md`. Treat unchecked tasks or old claims here as historical unless they are restated in the V9 docs.


V5 exposed the core problem:

- It created broad package, API, example, docs, and compatibility scaffolding.
- It still allowed visual proof that was not real renderer output.
- The release gallery could pass with canvas-painted scenes, simplified shapes, and proof images that did not come from a mature WebGL/WebGPU renderer rendering real imported assets.

V6 exists to stop that permanently.

This reset is not about making better reports. It is not about drawing prettier 2D canvas scenes. It is not about another set of stick figures, triangle cars, toy material grids, or fake visual parity screenshots.

V6 must build:

**A3D Production Renderer V6: a real WebGL2/WebGPU browser renderer that imports real assets, renders physically based materials with HDR image-based lighting, shadows, tone mapping, postprocess, animation, and controls, then proves same-scene quality against Three.js using real renderer pixels.**

The next product step is simple:

> A developer should be able to load real glTF/GLB assets, real HDR environments, real texture sets, and real production scenes in A3D and get renderer output that is credibly equal to or better than equivalent Three.js scenes for the documented scope.

V6 is not complete until A3D has real renderer output that can be inspected visually and measured technically.

## Product Objective

A3D must become a differentiated production renderer and developer product, not a clone of Three.js examples.

The V6 product must make a serious Three.js developer believe:

> "This is a real renderer library. It loads real assets, handles real PBR/HDR scenes, gives me high-quality defaults and workflow tooling, and can match or exceed Three.js quality for the documented production workflows."

V6 must create the foundation for differentiation:

- better production defaults than raw Three.js
- built-in real asset pipeline
- built-in HDR/IBL setup
- built-in physically based material validation
- built-in screenshot and visual QA gates
- built-in performance/profile diagnostics
- built-in same-scene migration and comparison tooling
- built-in product workflows for commerce, architecture, assets, and cinematic scenes

## Hard Truth From V5

The following V5 evidence is not acceptable as V6 completion proof:

- Canvas 2D painted "renderer" screenshots.
- Synthetic circles, blocks, stick figures, triangle cars, or primitive-only flagship visuals.
- Screenshots generated by `page.setContent()` with inline drawing scripts instead of running a real app.
- Tests that only verify `window.__app` JSON.
- Tests that only verify files exist.
- Pseudo renderer APIs that return strings such as capture URIs instead of real frame pixels.
- Visual parity scores hardcoded in scene metadata.
- Release gates that do not read image pixels or renderer frame buffers.
- "Three.js parity" reports that do not render the same scene in both engines.

V6 starts from this failure and must build gates that make it impossible to repeat.

Required failure documentation:

- `docs/project/production-runtime-roadmap-three-compat-visual-failure-audit.md`
- `docs/project/production-runtime-roadmap-no-fake-visual-proof.md`
- `tests/reports/production-runtime-three-compat-visual-failure-audit.json`

The audit must explicitly name the V5 failure modes:

- `tests/browser/three-compat-threejs-visual-parity.spec.ts` used canvas-painted visual evidence.
- `tests/reports/three-compat-gallery/*` was initially allowed to pass with tiny/blank or non-renderer screenshots.
- V5 release readiness was not enough because it did not require real WebGL/WebGPU renderer output from real app surfaces.

## Non-Negotiable Rules

V6 must obey these rules:

1. **Real renderer output only.**
   Release screenshots must come from A3D renderer frame buffers in WebGL2 or WebGPU, not Canvas 2D painting.

2. **Real imported assets only for flagship proof.**
   Flagship scenes must load real glTF/GLB assets and real texture/HDR files. Primitive-only scenes may exist as unit tests, but cannot count as flagship proof.

3. **No fake parity scores.**
   Visual parity must be computed from rendered screenshots and metrics. No `visualScore: 0.91` style metadata can decide pass/fail.

4. **No `page.setContent()` screenshot proof for flagship scenes.**
   Flagship screenshot tests must run actual apps/examples served through Vite or a static server.

5. **No Canvas 2D fallback for release screenshots.**
   Any flagship screenshot path generated from a 2D canvas context fails release readiness.

6. **No mock renderer as production proof.**
   Mock renderers can be used in unit tests only. Browser/release proof must use WebGL2 or WebGPU.

7. **No hidden fallbacks.**
   If WebGPU falls back to WebGL2, the report must say so. If a material extension is unsupported, the report must say so.

8. **No broad superiority claim without same-scene evidence.**
   A3D may target "equal or better than Three.js" only where same-scene comparison reports prove it.

9. **No completion with blank, flat, tiny, or low-entropy screenshots.**
   Pixel stats, structural similarity, histogram spread, edge density, object occupancy, and human review must all pass.

10. **No claim hiding.**
    Known gaps remain visible until implemented and verified.

## Hard Claim Boundary

V6 is allowed to claim only after release gates pass:

> "A3D V6 provides a production WebGL2 renderer with real asset loading, true PBR/HDR image-based lighting, shadows, postprocess, animation, controls, and same-scene Three.js visual parity for the documented flagship workflows."

V6 may claim WebGPU only as:

> "A3D V6 includes a WebGPU backend with documented feature coverage and parity gaps."

V6 is not allowed to claim:

- full Three.js API parity
- full Three.js ecosystem replacement
- full WebGPU parity unless every WebGPU parity gate passes
- Unity replacement
- Unreal replacement
- offline renderer parity
- every glTF extension
- every KTX2/Basis/UASTC path unless decoded and rendered
- performance superiority unless measured in same-scene benchmarks
