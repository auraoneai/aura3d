# Aura3D Product Boundaries

Version: 1.1.0 planning alignment

This document captures durable product rules extracted from legacy planning work. It is the current product boundary for Aura3D documentation, examples, templates, marketing, and agent instructions.

## Current Product Shape

Aura3D is a developer library, asset deployment pipeline, template system, diagnostics surface, and agent-readable documentation set for AI coding agents that write browser 3D app code.

The active authoring model is source code:

```ts
scene().add(model(assets.robot)).add(lights.studio());
```

The AI coding agent writes TypeScript or JavaScript against public Aura3D APIs. Aura3D does not ingest provider JSON as the primary scene-authoring contract.

## What Aura3D Owns

- public `@aura3d/engine` APIs;
- runtime defaults for common browser 3D scenes;
- typed GLB/glTF asset references generated by `@aura3d/cli`;
- scaffolded starter templates;
- scene kits and prefabs implemented on top of public primitives;
- diagnostics, screenshots, route health, and deployment checks;
- agent-readable docs, `llms.txt`, and setup files for coding assistants.

## What The Agent Owns

- translating the user's natural-language request into source code;
- choosing which public Aura3D APIs to use;
- composing scenes, interactions, UI, cameras, lighting, VFX, and animation;
- editing the app through normal source-code changes.

## What The User Owns

- product intent;
- source code;
- assets;
- licensing decisions;
- deployment target.

## Asset Pipeline And Catalog, Not Asset Generation

Aura3D supports both local asset registration and catalog-assisted discovery. The user or coding agent must still work with approved assets:

- `.glb`
- `.gltf`
- `.hdr`
- `.exr`
- `.png`
- `.jpg`
- animation clips
- audio

Aura3D starts after the asset exists on disk or after the CLI resolves a verified auto-pullable catalog candidate:

```bash
npx @aura3d/cli@latest assets search "animated humanoid fighting character" --profile fighting-character
npx @aura3d/cli@latest assets resolve "animated humanoid fighting character" --name fighter --profile fighting-character
npx @aura3d/cli@latest assets add ./assets/product.glb --name product
```

The CLI validates, fingerprints, copies or references, extracts metadata, preserves provenance where available, and generates typed imports. It does not purchase, create, host, or bless an asset as production-ready art. Game-character assets still require profile validation, visual review, licensing review, animation review, and route evidence.

## No Runtime Prompt-To-IR Product Claim

Legacy prompt-runtime work explored provider adapters, structured scene IR, server-side provider proxies, cinematic scene compilers, and provider-backed scene patching. That work is historical context unless it has been renamed, ported, documented, and verified as part of the current agent-written-code product.

Do not describe the current product as:

- a provider runtime;
- a prompt-to-IR compiler;
- a hidden scene generator;
- a server-side OpenAI, Anthropic, Gemini, or local-model proxy;
- a master cinematic asset universe;
- a consumer prompt-only 3D app.

Allowed replacement concepts:

- route evidence;
- diagnostics snapshots;
- typed asset metadata;
- prompt plans that produce source-code-oriented scene guidance;
- scene kits that agents call from TypeScript.
- a browser game-runtime foundation when the relevant runtime, asset, animation, and deployment gates have current evidence.

## No Release-Number Artifact Names

Do not name implementation artifacts after legacy release numbers or PRD labels. Names must describe what the artifact does.

Use names like:

- `cinematic-scene-quality`
- `product-viewer`
- `asset-workflow`
- `route-evidence`
- `prompt-to-3d-workflow`

Do not use names like:

- `v2-scene-runtime`
- `v3-renderer`
- `v4-agent-api`
- `Version3PromptLab`

Version labels can appear in changelogs, release notes, and historical prose, but not as the identity of developer-facing docs, routes, packages, templates, tests, tools, or exported APIs.

## Examples Teach Patterns

Examples prove API concepts. They do not imply Aura3D owns a content universe.

Examples should:

- use public APIs;
- teach one product or API concept at a time;
- link to source or route context;
- avoid duplicated scenes with different names;
- avoid implying marketplace content, cinematic assets, or product models are bundled inventory.

Templates are scaffolded projects. Examples are focused API proof. Do not blur those two roles in docs.

## API Stability Contract

Agent-written code depends on stable APIs.

Public `1.x` APIs must be documented before they appear in agent docs or templates. Breaking changes require migration notes. Deprecated APIs should remain available for at least one minor version unless there is a security or data-loss reason to remove them earlier.

Internal APIs must not appear in public examples, templates, marketing pages, or agent instructions.

## Aura3D advantage

Public claims must map to shipped APIs, templates, CLI commands, docs, examples, or current verification artifacts.

Do not use deleted PRDs, historical provider-runtime plans, or local-only report names as proof for public claims.
