# Examples And Benchmarks Plan

## Goal

Replace primitive proof pages with local examples and benchmarks that honestly show the engine's real capability. This is where the Three.js and Unity/Unreal-style claims either become credible in a narrow niche or remain blocked.

## Example Quality Bar

Every v3 flagship example must have:

- real model assets or generated assets with production-like complexity;
- orbit/pan/zoom or appropriate game/editor camera controls;
- visible lighting and shadows if the scene needs them;
- material/texture fidelity;
- nontrivial interaction;
- diagnostics panel that does not dominate the visual;
- browser screenshot evidence;
- Playwright interaction tests;
- known-limits section in page state.

## Metrics

Each benchmark must record:

- [ ] library version/commit;
- [ ] browser version;
- [ ] viewport and DPR;
- [ ] hardware class if available locally;
- [ ] bundle size;
- [ ] startup time;
- [ ] asset load time;
- [ ] first frame time;
- [ ] steady frame median/p95;
- [ ] draw calls;
- [ ] shader count;
- [ ] texture count/bytes;
- [ ] geometry count/bytes estimate;
- [ ] JS heap estimate if available;
- [ ] screenshot path;
- [ ] unsupported feature list.

## Claim Rule

A Three.js claim must be generated from benchmark data, not written manually. If Galileo3D loses or lacks feature parity, the report must say so.
